The word JavaScript conjures up pictures of dungeons and torture devices in my mind when I see it.  To me, it has always been an exercise in frivolity and futility, but sometimes it is necessary.  I'm going to give a few thoughts on what to do when that necessity rears its ugly head. 
Starting a project that requires vanilla JS always makes me shudder, but there are a few things I've learned that have made it a bit more barable.  As a Ruby on Rails dev, I function on convention over configuration, and I live by that mantra.  When the framework goes out the door, and we're required to ditch the niceties to which we are accustomed in the Rails world, that doesn't mean we can't keep the same philosophies we employ in Rails.  Here's one rule I've developed that applies nicely in either case.

1.  Keep it simple, stupid.
      
	I've found myself in situations very often where I code myself into a corner.  I tend to get a bit overzealous with the code, and it gets me in trouble.  This doesn't have to be the case with good planning.  Know what you want to build, and build only that.  This will allow you to start from the beginning, and end at the end.  No one wants their project flow to start and stop and restart over and over again.  If you have a good idea of where you need to go from the beginning, you'll have an easier time getting there.
	
	
	
	![](https://66.media.tumblr.com/bfc2622a2dbe6ec766177bc86cafa534/tumblr_pgtnzgpAKy1qfn2zzo2_540.gif/)
	
	
	The best way to do this is to keep things simple.   If there is a possibility that your code will require the creation of a new Class, create that stinking class before you realize you can't shove its logic into something that already exists.  Don't build anything that doesn't directly affect functionality or UX unless it was previously planned.
	
	
	Learn how you work, and it will make a world of difference.
